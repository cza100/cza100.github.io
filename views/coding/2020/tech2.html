# 前言

服务器越来越多，但每次部署、升级程序都要靠手动操作。虽然得益于 Docker 的使用，大大减少了部署的复杂性，但构建一套运维系统还是很有必要的。经过调研，决定使用 Ansible 作为运维工具。

# 初次尝试

因为 Ansbile 上手比较简单，只要搞清 Inventory、Task、Playbook 这几个概念就可以快速使用，等遇到具体的运维需求，再去找相应的模块编写 playbook 就可以了。官方也对 ansbile 的目录结构给出了最佳实践，至于 Role 的使用还是看个人吧，我是直接以 tag 区分任务。

# 整合到现有后台系统

通过命令行操作 ansible 肯定是不行的，看了官方的 UI Ansible Tower 好像是要收费的，而且我们本来就有一套内部的管理系统，就决定将 ansible 整合到这个系统里。

我们的后台系统用的是 laravel + swoole，所以准备使用 PHP 的shell_exec来执行 ansible 命令，并用 swoole 的async task来执行任务队列。

关于 Inventory,我这边的话直接使用 php file_put_content函数将数据库中的 Inventory 数据写入 Inventory 文件。

foreach($groups as $group) {
  file_put_contents('/etc/ansible/hosts', '[' . $group->value . ']' . PHP_EOL, FILE_APPEND | LOCK_EX);
  $hosts = $group->ecs()->get();
  if ($hosts) {
    foreach($hosts as $host) {
      file_put_contents('/etc/ansible/hosts', $host->hostname . ' target_os=' . $host->os . PHP_EOL, FILE_APPEND | LOCK_EX);
    }
  }
}

下面是shell_exec调用 ansible 的接口函数：

if (!function_exists('call_ansible')) {
  function call_ansible($cmds, $group = '', $targets = [], $extra = '') {
    $cmd_section = '--tags ' . '"' . implode(',', $cmds) . '" ';

    $limit_section = '';
    if ($group) {
      $limit_section = '--limit ' . $group;
    } else {
      $limit_section = '--limit ' . '"' . implode(',', $targets) . '" ';
    }

    $glue = 'ansible-playbook ' . '/root/playbooks/' . 'main.yml ' . $cmd_section . $limit_section . $extra;
    $play_res = shell_exec($glue);
    return $play_res;
  }
}

# 发现问题

刚开始设计时没有考虑到长时间运行的任务，比如说 docker images 的拉取等。发现运行 ansible 任务必然是要阻塞的。后来使用 ansible 的 async 和 polling，但发现任务进行即使异步了，但是想要获取任务运行的结果的话，当前的shell_exec执行的 shell 命令还是阻塞的。那等于说是 swoole 任务队列不会进行下一个任务。

由于服务器配置极低(1 核 1G)，所以 swoole task 数量只开了 1。所以碰到这种长时间任务，只能堵塞了。其实这种情况，开多个进程 worker 其实也可以解决问题，但 ansible 在返回信息格式上存在缺陷，导致我决定重新设计模式。

# 重新设计

经过一番思索，决定用自定义脚本的方式来完成运维。简单来说，ansible 只作为一个传递脚本的中间角色，脚本传递到目标服务器独立运行，运行完成后主动向后台上报结果。

优点：

  • 每个运维任务可直接在后台网页端编辑，更加灵活自由。
  • 执行长时间任务完全独立运行，不会阻塞。

缺点：

  • 执行一系列任务要控制顺序，要实现触发的逻辑。
  • 一些基本信息获取（系统版本等）、条件判断、循环不如直接使用 ansible 的 playbook 方便。
  • 每个脚本都要编写上传结果的额外代码（当然可以通过设计一个类似于代码片段的方式来复用）。

流程：

1. 后台编写任务脚本
2. 通过ansible传递脚本至目标
3. 目标执行脚本
4. 目标上传执行结果
5. 数据库保存结果

# UUID 的传递

每个脚本执行完，要上传结果，那怎么知道自己任务的 uuid 呢？

直接通过 ansible 的 extra-vars 来传递 uuid 值，在 shell 脚本中就可以通过$1来获取。

$extra_vars = [
  'uuid' => $task->uuid,
  'cmd' => $cmd->value,
  'lang' => $cmd->lang,
];

$extra_vars = json_encode($extra_vars);
costum_ansible($extra_vars, $group, $targets);


if (!function_exists('costum_ansible')) {
  function costum_ansible($extra_vars, $group = '', $targets = []) {
    $extra_section = '--extra-vars ' . "'" . $extra_vars . "'";

    $limit_section = '';
    if ($group) {
      $limit_section = '--limit ' . $group . ' ';
    } else {
      $limit_section = '--limit ' . '"' . implode(',', $targets) . '" ';
    }

    $glue = 'ansible-playbook ' . '/root/playbooks/' . 'cmd.yml ' . $limit_section . $extra_section;
    \Log::info($glue);
    $play_res = shell_exec($glue);
    return $play_res;

  }
}

# 支持 python 脚本

python 毕竟也可以作为一门运维语言，有时候比 shell 间接多了。那么可否传递 python 脚本来执行任务呢？

这边我直接通过第一行注释来决定执行 shell 还是 python,当第一行为#!/bin/bash时，传递给 ansible 时就加个 lang 为 shell 的额外参数，python 同理。

ansible 执行时判断脚本类型：

- name: run costum tasks
  shell:
    cmd: |
      sed -i 's/\r$//' {{ cmd }} && {{ lang }} {{ cmd }} {{ uuid }}
    chdir: /root/commands/

# 依赖文件的传递

有时候，执行任务需要一些依赖的安装包，再通过脚本去获取有些麻烦，所以我直接用 ansible 来拷贝依赖文件。

这里自定义了一个文件拷贝的语法：

//在脚本的第二行
#files=files/version.yml,files/test.yml
1
2

这样 ansible 就会把文件拷贝到目标执行脚本的目录。脚本中也直接可以通过当前目录来操作文件。

ansible 中执行拷贝额外参数中的文件

- name: copy require files
  copy:
    src: "{{ item }}"
    dest: /root/commands/
  with_items: "{{ files }}"

# 任务执行顺序

我采用的方式是类似于反向的单向链表，即下一个任务保存上一个任务的 UUID，当任务完成后，会寻找有没有任务保存了自己的 UUID，找到那个任务并执行。如下：

uuid before_uuid
a73501 null
a73502 a73501

当 uuid 为 a73501 的任务执行完，会寻找 before_uuid 为 a73501 的任务并执行。
